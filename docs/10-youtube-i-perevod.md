# YouTube: обход бот-чека и перевод озвучки

Этот раздел про две связанные вещи, добавленные поверх базовой качалки: закадровый перевод видео силами Яндекса и обход защиты YouTube "Sign in to confirm you're not a bot". Обе опираются на неофициальные, реверс-инженеренные сервисы, поэтому тут же расписано, как это сопровождать, когда сломается.

## Перевод озвучки через Яндекс

### Что это делает

Яндекс-Браузер умеет переводить закадровый голос иностранных видео на русский. Тот же движок (VOT, Video Translation) доступен через сторонний CLI `vot-cli`. Качалка использует его, чтобы отдать пользователю видео с русской озвучкой поверх оригинала.

Перевод переводит именно **иностранную** речь в русскую. Русскоязычное видео Яндекс не тронет - переводить нечего.

### Как включить

Перевод живёт в меню долгого удержания кнопки "Скачать" (там же, где выбор 4K/2K/1080p) - пункт "Перевод 🇷🇺". Он не тумблер: при выборе ставится скрытое поле `translate`, качество остаётся дефолтным. Только для видео, для аудио пункт бессмыслен.

### Пайплайн

Одна фоновая задача, обёрнутая в `bash -c`, три шага через `&&`:

1. **yt-dlp** качает ролик как обычно (через прокси или напрямую, в зависимости от домена). Флаг `--print-to-file after_move:filepath` пишет путь готового файла во временный `vpath`.
2. **vot-cli** по ссылке ролика тянет с серверов Яндекса готовую русскую аудиодорожку (Яндекс сам скачивает видео у себя, поэтому работает только для публичных сайтов, что Яндекс знает - YouTube, VK, Vimeo и т.п.).
3. **ffmpeg** (`app/mux_translated.sh`) вклеивает перевод первой дорожкой (default), оригинал оставляет второй. Итог - `*_ru.mp4`, исходный файл удаляется.

`&&` между шагами даёт мягкую деградацию: если перевод не удался (Яндекс не знает сайт, видео уже русское, сервис отвалился), mux не запускается и остаётся обычное скачанное видео, а не пустая ошибка.

### Нормализация ссылок

Яндекс-VOT капризен к форме URL. `Downloader::executeDownload()` перед вызовом vot-cli приводит ссылку к понятному виду:

- `youtube.com/shorts/ID` → `youtube.com/watch?v=ID`
- `vkvideo.ru/video...` (и `vk.ru`, `vkvideo.com`) → `vk.com/video...`

yt-dlp качает оригинальную форму - правится только копия URL для vot-cli.

### Статус в интерфейсе

Пока идёт перевод, задача в "Загрузках" показывает не безликое "В Процессе", а фазу: "Перевожу озвучку через Яндекс" (пока работает vot-cli) и "Вклеиваю русскую дорожку в видео" (пока mux'ит ffmpeg). Фазы ловятся по маркерам в логе (`Response language is set to`, `[vot]`, `frame=`), которые бывают только у задач с переводом.

### Сопровождение vot-cli

`vot-cli` ставится глобально через npm в `Dockerfile`. Он ходит в приватный API Яндекса общим для всех пользователей HMAC-ключом. Когда Яндекс меняет ключ, перевод отваливается **у всех**, пока мейнтейнер vot-cli не выпустит обновление.

- **Симптом**: любой перевод даёт "unknown service" или "перевод не найден", видео при этом качается нормально, но без русской дорожки.
- **Проверить вручную** (подставь реальный id):
  ```bash
  docker exec yt-prod_dev vot-cli --reslang=ru 'https://www.youtube.com/watch?v=<id>'
  ```
  Покажет, переводит Яндекс этот ролик или нет.
- **Починить**: пересобрать образ (`./build.sh build dev no-cache`) - подтянет свежий `vot-cli`. Либо на горячую, без пересборки: `docker exec -u root yt-prod_dev npm install -g vot-cli`.

## Обход бот-чека YouTube (bgutil PO-token)

### Зачем

YouTube требует PO-токен - доказательство, что запрос идёт от живого клиента, а не скрипта. Без него, особенно через засвеченный прокси-IP, yt-dlp получает `HTTP 429` и `ERROR: Sign in to confirm you're not a bot`. Скачивание не идёт.

Решение - `bgutil-ytdlp-pot-provider`: он генерирует PO-токены локально, без кук и без аккаунта Google. Токен решается в JS-песочнице (BotGuard), браузер и Chromium не нужны.

### Как устроено

Две части:

1. **yt-dlp-плагин** - ставится через pip в `Dockerfile`. yt-dlp находит его сам в `site-packages` (пространство имён `yt_dlp_plugins`) и на каждой YouTube-загрузке запрашивает токен у сервера.
2. **Сервер-генератор** - Node/TypeScript, собирается из git при сборке образа в `/opt/bgutil`, запускается фоном в `start.sh`, слушает `127.0.0.1:4416`.

В команде yt-dlp за это отвечают две вещи, добавленные в `executeDownload()`:

- `--plugin-dirs default` рядом с `--plugin-dirs <каталог лог-плагина>`. **Это критично.** В нынешнем yt-dlp `--plugin-dirs` *заменяет* дефолтные пути поиска, а не дополняет. Без `default` каталог лог-плагина вытесняет `site-packages`, и pip-плагин bgutil молча не грузится - YouTube снова режет бот-чеком. Если после апдейта yt-dlp бот-чек вернулся - первым делом проверь, что `default` на месте.
- `no_proxy=127.0.0.1,localhost` в переменных окружения проксируемых загрузок. Иначе запрос yt-dlp к локальному серверу токенов ушёл бы в SOCKS5-прокси и не дошёл.

### Сопровождение bgutil

Плагин (pip) и сервер (сборка из git) должны быть **одной версии** - они релизятся в связке. Пересборка образа тянет свежие обе. Когда YouTube меняет BotGuard, bgutil выпускает патч, ты пересобираешь.

- **Симптом**: в "Последних" снова "YouTube принял нас за бота 🤖".
- **Проверить сервер живой**:
  ```bash
  docker exec yt-prod_dev sh -c 'curl -s 127.0.0.1:4416/ping; echo; tail /var/www/YT/tmp/bgutil.log'
  ```
  `/ping` должен вернуть JSON с версией.
- **Проверить, что плагин грузится**:
  ```bash
  docker exec yt-prod_dev /.yt_env/bin/yt-dlp --plugin-dirs default -v --simulate 'https://youtu.be/<id>' 2>&1 | grep -i pot
  ```
- **Починить**: `./build.sh build dev no-cache`.

### Важные грабли

- Сервер bgutil в `start.sh` **без авто-рестарта**. Если он умрёт, контейнер останется жив (nginx держит его), но бот-чек вернётся. При внезапных бот-чеках первым делом смотри `/ping` и `bgutil.log`.
- yt-dlp обновляется хостовым cron'ом (`app/etc/Scripts/update-ytdlp.sh`). Он переприменяет патч имён файлов, но bgutil и vot-cli не трогает. Крупный апдейт yt-dlp теоретически может сменить интерфейс PO-токена - тогда обновляй и bgutil.

## Что влияет на скорость бана IP

Отдельный больной вопрос: какие настройки и действия приближают бан или `429` от YouTube. Разбор по убыванию влияния.

### 1. Лимит одновременных загрузок (`max_dl`)

Главный рычаг. `config.php`, ключ `max_dl`. Сейчас **1** - это правильно и безопасно: в любой момент к сайтам стучится один процесс yt-dlp.

- `max_dl => -1` (без лимита) - **худшее, что можно сделать**. Пачка ссылок разом поднимет столько процессов, сколько ссылок, все долбят с одного IP. Прямой путь к бану.
- Не поднимай выше 1-2, если качаешь много с YouTube.

### 2. Тонкость: параллельность обходит `max_dl`

Важно понимать: `max_dl` проверяется **один раз на отправку формы**, а не на процесс. Внутри `addOneDownload()` одна отправка может породить несколько параллельных процессов yt-dlp:

- Ссылки группируются по хосту. Разные хосты - разные процессы, идут параллельно (это ок для бана: у каждого сайта по одному подключению).
- Несколько ссылок **одного** хоста (например, 10 ссылок YouTube через `||`) бьются на два процесса: первая ссылка качается сразу, остальные - вторым процессом с паузами. То есть к YouTube в моменте два подключения с твоего IP.

Вывод: `max_dl=1` не значит "один процесс к YouTube всегда". Пачка YouTube-ссылок за раз - это до двух параллельных потоков к нему. Не катастрофа, но и не один.

### 3. Список: плейлист против пачки ссылок

Есть два способа скачать много роликов, и они **очень** разные по риску:

- **Ссылка на плейлист/канал** (`youtube.com/playlist?list=...`, `/channel/`, `/@handle`) - самый безопасный. yt-dlp разворачивает её в десятки роликов **внутри одного процесса**, последовательно, с паузами. Качалка это распознаёт (`isYoutubeMulti`) и вешает `--sleep-requests 1.5 --sleep-interval 3 --max-sleep-interval 8`. Один поток, вежливые задержки.
- **Много отдельных ссылок через `||`** - хуже. Как описано выше, это два параллельных процесса к YouTube, и пауза есть только у второго ("хвоста"). Первая ссылка стартует без задержки.

**Рекомендация**: качаешь канал или подборку - давай Качалке ссылку на плейлист/канал, а не вставляй десятки отдельных ссылок. Один вежливый поток безопаснее двух торопливых.

### 4. Одиночные ролики без пауз

Одиночное видео идёт **без** `--sleep`, потому что делает мало запросов - навешивать паузы смысла нет, юзер только злится на "Пауза N сек" на пустом месте. Но если руками быстро скармливать много одиночных ссылок подряд - каждая создаёт всплеск запросов к player API. При `max_dl=1` они хотя бы встают в очередь и идут по одной, но back-to-back, без пауз между задачами. Массовый ручной спам одиночками - реальный способ словить `429`.

### 5. Параллельные фрагменты (`--concurrent-fragments 4`)

Для проксируемых загрузок (включая YouTube) стоит `--concurrent-fragments 4 --http-chunk-size 5M` - четыре параллельных Range-запроса к CDN. Это осознанный компромисс (4, не 8): скорость упирается в прокси, а меньше параллельных запросов к CDN с одного IP - меньше шанс словить 403 на чанк. Трогать без нужды не стоит; поднимать до 8+ - повышать заметность.

### 6. Репутация IP - корень всего

Самое важное и то, что кодом не лечится. `429` на **первом же** запросе webpage (видно в логах) означает, что твой SOCKS5-IP уже зарейтлимичен на уровне IP. Пейсинг это смягчает, но не устраняет: датацентровый прокси-IP YouTube узнаёт и прижимает независимо от того, как вежливо ты качаешь.

- bgutil снимает **бот-чек**, но не сырой rate-limit по IP.
- Реально надёжное лекарство от `429` - **резидентный прокси** вместо датацентрового. Это деньги/инфраструктура, но никакая настройка yt-dlp не заменит чистый IP.

### Короткий чек-лист "не бань меня быстро"

- `max_dl` держи 1 (максимум 2). Никогда не `-1`.
- Много роликов - ссылкой на плейлист/канал, не пачкой отдельных ссылок.
- Не спамь одиночными ссылками в темпе пулемёта.
- `--concurrent-fragments` не задирай.
- Если `429` лезет постоянно - проблема в IP прокси, а не в настройках. Думай про резидентный прокси.
