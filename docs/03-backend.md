# Backend: разбор кода

Backend написан на чистом PHP, без фреймворков. Весь код сосредоточен в нескольких файлах: точка входа `index.php` и три класса в папке `class/`.

## index.php - точка входа

Этот файл обрабатывает каждый запрос к сайту. Внутри он делает несколько вещей по очереди.

**Настройка сессии и заголовков.** Перед стартом сессии файл настраивает cookie (httponly, samesite, secure при HTTPS) и выставляет заголовки безопасности. Подробный разбор - в [Безопасности](08-security.md).

**Проверка CSRF.** Две функции, `generateCsrfToken()` и `validateCsrfToken()`, защищают от подделки запросов. Токен генерируется один раз на сессию и сверяется при каждом действии, которое что-то меняет (запуск загрузки, удаление файла, остановка процесса).

**Отдача статуса (`?jobs`).** Если в запросе есть параметр `jobs`, сайт не рисует страницу, а просто отдаёт JSON со всем, что нужно фронтенду: список активных загрузок, очередь, завершённые загрузки и списки файлов (видео и музыка). Именно этот эндпоинт JavaScript дёргает каждые 1.5-12 секунд.

**Обработка действий (POST).** Дальше идёт блок, который смотрит, что пришло в POST-запросе:
- `urls` - запустить новую загрузку. Перед созданием объекта `Downloader` проверяется, что на диске есть минимум 100 МБ свободного места (через `FileHandler::get_free_space_bytes()`). Если места недостаточно, пользователю показывается ошибка вместо попытки загрузить, что предотвращает ситуацию, когда yt-dlp упадёт с "No space left on device" уже в процессе.
- `kill` - остановить одну или все загрузки (`kill=all`).
- `delete` - удалить готовый файл с диска.
- `clear` - почистить историю (`recent`), очередь (`queue`) или одну запись.
- `restart` - перезапустить ранее отменённую или завершённую (в том числе с ошибкой) загрузку с той же командой.
- `removeQueued` - убрать конкретную задачу из очереди, пока она там ещё ждёт.

Все эти действия объединены в одну проверку `$isDestructive` - если запрос что-то меняет, сначала проверяется CSRF-токен, и если он не совпал, показывается страница ошибки из `error_pages.php`.

**Доверие к IP клиента.** Перед запуском загрузки `index.php` определяет настоящий IP пользователя. Заголовки `X-Forwarded-For` и `X-Real-IP` подделать может кто угодно, поэтому им доверяют только если запрос пришёл с приватного адреса (то есть точно через свой nginx или докер-сеть, а не прямо из интернета). Иначе используется `REMOTE_ADDR` - тот адрес, который видит сам сервер.

## class/Downloader.php - главный класс

Это самый большой и важный файл. Он отвечает за всё: запуск yt-dlp, чтение статусов, разбор ошибок, очередь, остановку процессов.

### Список доменов прямого доступа

В начале файла объявлен список `DIRECT_ACCESS_DOMAINS` - российские и не заблокированные сайты (vk.com, rutube.ru, ok.ru, телеканалы, Яндекс и так далее). Загрузки с этих доменов идут напрямую, без прокси - потому что иностранные прокси часто либо блокируются такими сервисами, либо просто работают нестабильно. Все остальные сайты (включая YouTube) идут через SOCKS5-прокси, если он настроен.

### Конструктор

При создании объекта `Downloader` с непустым списком ссылок он:
1. Проверяет, что папка для загрузок существует и доступна для записи.
2. Разбивает каждую ссылку по разделителю `||` (так можно скачать сразу несколько видео одной отправкой формы) и проверяет каждую через `filter_var(..., FILTER_VALIDATE_URL)`.
3. Если все ссылки валидны - запускает `do_download()`.

### do_download() и addOneDownload()

`do_download()` смотрит на настройку `max_dl` из конфига:
- `-1` - лимита нет, запускать сразу.
- больше `0` - если текущих фоновых загрузок меньше лимита, запускать сразу, иначе (если очередь не отключена) поставить в очередь.

`addOneDownload()` разбивает список ссылок на две группы - те, что идут через прокси, и те, что идут напрямую (по списку доменов выше) - и вызывает `executeDownload()` отдельно для каждой группы.

### executeDownload() - сборка команды yt-dlp

Здесь собирается итоговая команда, которая запустится в shell. По шагам:

- Базовый бинарник из конфига плюс `--js-runtimes node` (см. [Архитектуру](01-architecture.md)).
- Подключение Python-плагина логирования через `--plugin-dirs` и `--use-postprocessor`.
- Шаблон имени файла и `--restrict-filenames`.
- Выбор качества. Если выбран "Булшит" (worst) - просто `-f worst`. Иначе видео качается до потолка `maxVideoRes` из конфига, а если пользователь зажал кнопку "Скачать" и выбрал явное качество (4K, 2K, 1080p) - это качество перекрывает потолок из настроек.
- Если используется прокси: добавляются `--concurrent-fragments 8 --http-chunk-size 5M`. Смысл этих флагов: видеохостинги вроде YouTube часто отдают видео одним файлом с поддержкой докачки по диапазонам (Range), а не нарезанным на готовые фрагменты (как HLS/DASH). Эти флаги режут такой файл на куски по 5 МБ и качают несколько кусков параллельно - это сильно ускоряет скачивание через медленный прокси.
- Если выбрана конвертация в аудио - добавляется `-x` и формат (mp3, wav, aac, flac).
- Каждый параметр, который приходит от пользователя (URL, прокси, путь), оборачивается в `escapeshellarg()` - это защита от command injection, подробнее в [Безопасности](08-security.md).
- Команда запускается через `exec()` с перенаправлением вывода в файл и сохранением PID процесса (`echo $! > pid-файл`), чтобы потом можно было найти и остановить именно этот процесс.

### get_current_background_jobs() и get_finished_background_jobs()

Это пара методов, которые читают файлы из `tmp/` и превращают их в массив данных для фронтенда.

`get_current_background_jobs()` идёт по всем файлам `pid_*`, проверяет, жив ли ещё процесс (через `/proc/$pid`), и если жив - читает соответствующий лог-файл yt-dlp и вытаскивает из него текущий статус (название файла, на каком сайте, процент загрузки). Если процесс уже умер - лог переименовывается в `ytdl_*` (считается завершённым), а команда дополнительно проверяется, что это точно процесс yt-dlp (через чтение `/proc/$pid/cmdline`), а не случайно совпавший PID от другого процесса.

`get_finished_background_jobs()` делает похожее, но для уже завершённых логов (`ytdl_*`). Если в логе так и не нашлось имя файла - значит, что-то пошло не так, и запускается `parseYtDlpError()`, чтобы понять, что случилось.

### parseYtDlpError() - перевод ошибок на человеческий язык

Метод получает текст лога yt-dlp и через десятки регулярных выражений ищет знакомые паттерны ошибок: не резолвится DNS, сайт вернул 403 или 404, видео приватное, требует авторизации, гео-блок, нет форматов для скачивания, не хватает места на диске и так далее. Каждому типу соответствует короткое русское сообщение с эмодзи. Если ничего не подошло - в крайнем случае вытаскивается сырой текст строки `ERROR:` из лога yt-dlp.

Перед сравнением лог обязательно прогоняется через `sanitizeLog()` - это важно, потому что в логе могут быть IP-адреса, пароли от прокси или другие чувствительные данные, и показывать их пользователю нельзя.

### process_queue() - продвижение очереди

Очередь хранится в одном файле `dl_queue`, где каждая строка - это одна ожидающая загрузка. Метод открывает файл, ставит эксклюзивную блокировку (`LOCK_EX`), читает все строки, и для тех, на кого хватает места по лимиту `max_dl`, запускает реальную загрузку. Остальные строки остаются в файле. Блокировка нужна, чтобы два параллельных запроса (например, два человека одновременно открыли сайт) не испортили файл очереди друг другу.

### kill_one_of_them() и kill_them_all()

Останавливают один или все процессы yt-dlp. Важная деталь: перед тем как слать `kill`, метод проверяет через `/proc/$pid/cmdline`, что это действительно процесс yt-dlp, а не случайно совпавший по номеру чужой процесс на сервере. `kill_them_all()` ищет все процессы по имени (`pgrep -f 'yt-dlp'`) и завершает их, а также удаляет недокачанные `.part`-файлы, если в конфиге не включено `keepPartialFiles`.

### restart_download()

Перезапускает старую загрузку, читая сохранённую команду прямо из лога. Здесь есть отдельная проверка безопасности: команда из лога сверяется с ожидаемым путём к `yt-dlp` - если она не совпадает, рестарт отменяется, чтобы кто-то не мог подменить лог-файл и заставить сервер выполнить чужую команду.

Опциональный параметр `$forceUseProxy` (по умолчанию `false`) позволяет при ретрее принудительно добавить SOCKS5 прокси, даже если исходная задача запускалась напрямую. Это используется системой авторетрея при гео-блоке/403.

### autoRetryIfNeeded() и isRetryableError()

Когда задача завершается, `get_current_background_jobs()` вызывает `autoRetryIfNeeded()`. Метод проверяет:
1. Есть ли в логе маркер `[RETRY_ATTEMPTED:]` - если есть, авторетрей заблокирован (уже был один раз).
2. Парсит ошибку через `parseYtDlpError()` и вызывает `isRetryableError()` для определения типа ошибки.
3. Если ошибка сетевая (тайм-аут, 429, 503, DNS ошибки) - добавляет маркер `[RETRY_ATTEMPTED:timestamp]` в лог и вызывает `restart_download($fpid, true)`.

Метод `isRetryableError()` отличает временные сетевые ошибки (стоит ретрейть) от постоянных проблем контента (не стоит ретрейть). Это полезно когда сайт блокирует прямые подключения (403 Forbidden, гео-блок), но пропускает через SOCKS5 прокси.

Для пользователя авторетрей полностью прозрачен: если ретрей удался, он видит только готовый файл; если оба запуска упали, он видит финальную ошибку.

## class/FileHandler.php - работа с файлами

Этот класс знает только про файлы на диске - он не запускает yt-dlp и не лезет в процессы.

- `listVideos()` и `listMusics()` - сканируют папку загрузок и возвращают файлы с нужными расширениями, плюс возраст файла в минутах и процент "оставшегося времени жизни" (для таймера на 2 часа).
- `delete()` - удаляет файл по имени. Здесь стоит защита от выхода за пределы папки: имя файла сначала чистится через `basename()` (убирает любые `../`), а затем через `realpath()` проверяется, что итоговый абсолютный путь действительно находится внутри разрешённой папки загрузок.
- `to_human_filesize()` - превращает байты в читаемый размер (КБ, МБ, ГБ).

## class/Session.php

Маленький класс-синглтон (это паттерн, который гарантирует, что объект создаётся только один раз за всё время работы программы) для запуска сессии. На самом деле сейчас этот класс никем не вызывается - сессия стартует напрямую через `session_start()` в `index.php`. `Session.php` выглядит как заготовка на случай, если в проекте появится несколько точек входа, но пока это мёртвый код, который можно было бы убрать.

## error_pages.php

Содержит одну функцию `showCsrfErrorPage()` - красивую страницу с кодом 403, которая показывается, если проверка CSRF-токена не прошла (например, страница была открыта слишком давно и токен в сессии устарел).
