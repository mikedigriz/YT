# Архитектура

## Общая картина

Качалка - это классическое PHP-приложение без фреймворков, плюс фоновые процессы yt-dlp и небольшой JavaScript на фронтенде. Никакой базы данных нет - вся информация о текущих загрузках, очереди и истории хранится прямо в файлах на диске.

Путь одного запроса на скачивание выглядит так:

1. Пользователь вставляет ссылку в поле на главной странице и жмёт "Скачать".
2. Браузер отправляет POST-запрос в `index.php`.
3. `index.php` проверяет CSRF-токен, разбирает параметры (аудио или видео, какой формат) и создаёт объект `Downloader`.
4. `Downloader` решает: запускать загрузку прямо сейчас или поставить в очередь (если уже работает максимум разрешённых загрузок).
5. Если можно запускать - `Downloader::executeDownload()` собирает командную строку `yt-dlp` и запускает её в фоне через `exec()`. PHP не ждёт, пока видео скачается - он сразу возвращает пользователя на страницу.
6. yt-dlp работает в фоне и пишет собственный прогресс в текстовый файл-лог. Параллельно специальный Python-плагин (`LogPluginPP`) дописывает каждую завершённую загрузку в общий журнал `/var/log/yt_dlp.log`.
7. Браузер раз в 1.5-12 секунд дёргает `index.php?jobs` и получает JSON с текущим статусом всех загрузок, очереди и списком готовых файлов.
8. JavaScript на странице перерисовывает таблицы по этим данным - без перезагрузки страницы.
9. Когда файл готов, он лежит в папке `download/`. Через 2 часа отдельный cron-скрипт удаляет старые файлы.

## Компоненты

- **nginx** - принимает запросы, отдаёт статику (картинки, css, js), всё остальное передаёт в php-fpm. Заодно блокирует прямой доступ к служебным папкам (`tmp`, `class`, `config`). Это внутренний nginx, который живёт прямо в контейнере. В продакшене перед контейнером обычно ставится ещё один, внешний nginx на хосте - он занимается доменом, HTTPS и защитой от ботов, подробнее в [Внешнем nginx перед Docker](09-reverse-proxy.md).
- **php-fpm** - выполняет `index.php` и классы из `class/`.
- **yt-dlp** - сама программа для скачивания видео. Запускается как отдельный процесс в фоне, никак не связанный с жизненным циклом PHP-запроса.
- **Python-плагин LogPluginPP** - подключается к yt-dlp как постпроцессор и пишет общий журнал загрузок (кто, что и когда скачал).
- **JavaScript на фронтенде** - опрашивает сервер и обновляет интерфейс в реальном времени, без участия сервера в рендеринге HTML после первой загрузки страницы.

## Почему yt-dlp требует Node.js

Если посмотреть на команду запуска yt-dlp в `Downloader::executeDownload()`, там есть флаг `--js-runtimes node`. Это не случайность.

YouTube в 2024-2025 годах начал защищать ссылки на видео через JS-загадки: чтобы получить настоящий адрес видеопотока, нужно выполнить определённый JavaScript-код прямо на странице (он вычисляет так называемый n-параметр и PoToken). Без выполнения этого кода YouTube либо не отдаёт ссылку, либо сильно тормозит скачивание. Подробности этой защиты и борьбы с ней можно найти в issues и changelog upstream-проекта [yt-dlp на GitHub](https://github.com/yt-dlp/yt-dlp).

yt-dlp решил эту проблему так: теперь он может запускать настоящий JavaScript-движок (Node.js, Deno или встроенный движок) и выполнять JS-код YouTube как обычный браузер. Поэтому в `Dockerfile` проекта:

- Ставится Node.js версии 22 и выше из репозитория NodeSource (версия из стандартного репозитория Debian слишком старая).
- Устанавливается `yt-dlp[default]` - расширенная версия с пакетом `yt-dlp-ejs`, который содержит сами решатели этих JS-загадок. Без него Node 22 сам по себе ничего не разблокирует - нужен именно этот набор скриптов.
- В команде запуска yt-dlp прописан флаг `--js-runtimes node`, который явно указывает: используй Node.js для выполнения JS-кода.

Без этой связки (Node 22 + yt-dlp-ejs + флаг) скачивание с YouTube либо не работает, либо постоянно упирается в ошибки про недоступные форматы.

## Как хранится состояние загрузок

Никакой базы данных в проекте нет. Вместо неё - простая система файлов в папке `tmp/` (путь задаётся в `config.php` параметром `logPath`):

- `pid_*` - файл с PID запущенного процесса yt-dlp, командой запуска и URL. Существование такого файла означает "загрузка идёт".
- `job_*` - лог самого процесса yt-dlp (то, что он пишет в консоль).
- `ytdl_*` - переименованный `job_*`-файл после завершения загрузки. Означает "загрузка закончена, лог сохранён".
- `dl_queue` - один файл, где построчно хранятся все ожидающие своей очереди загрузки.

Когда нужно узнать список активных загрузок, `Downloader::get_current_background_jobs()` просто читает все файлы `pid_*` и проверяет, существует ли такой процесс в `/proc/$pid` - если процесса нет, значит загрузка завершилась, и её лог переименовывается в `ytdl_*`.
