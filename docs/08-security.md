# Безопасность

Качалка запускает на сервере внешние процессы по ссылкам, которые присылают пользователи из браузера - в такой задаче легко получить дыру для атаки, если не думать о защите заранее. Вот что реально сделано в проекте.

## Защита от CSRF (подделки запросов)

Каждой сессии выдаётся случайный токен (`generateCsrfToken()` в `index.php`, 32 случайных байта через `random_bytes()`). Любое действие, которое что-то меняет - запуск загрузки, остановка процесса, удаление файла, очистка истории, перезапуск - требует этот токен в запросе, и сверка идёт безопасным сравнением `hash_equals()`, которое защищено от атак по времени отклика.

Дополнительно проверяется заголовок `Origin` или `Referer` - если запрос пришёл не с того же домена, он тоже отклоняется. Если проверка не прошла, показывается отдельная страница ошибки (`error_pages.php`) с понятной инструкцией: обновить страницу и попробовать снова.

На фронтенде токен сам подставляется в каждую форму перед отправкой - пользователю не нужно ничего делать вручную, и токен невозможно забыть отправить случайно.

## Защита от command injection

Качалка запускает yt-dlp через `exec()` с командной строкой, собранной из пользовательских данных (URL, формат, путь). Это самое опасное место в любом таком проекте - если строку собирать неаккуратно, пользователь может вставить в поле ссылки что-то вроде `; rm -rf /` и заставить сервер выполнить произвольную команду.

От этого защищает `escapeshellarg()` - каждый параметр, который может прийти от пользователя, оборачивается в эту функцию перед попаданием в командную строку. Она гарантирует, что даже если в URL есть кавычки, точки с запятой или другие специальные символы shell, они останутся просто частью текста, а не превратятся в отдельную команду.

## Защита от path traversal при удалении файлов

Когда пользователь просит удалить файл, имя файла сначала прогоняется через `basename()` - эта функция убирает любые попытки выйти за пределы папки через `../`. Дальше `FileHandler::delete()` дополнительно проверяет реальный путь через `realpath()` и убеждается, что итоговый файл физически находится внутри разрешённой папки загрузок, а не где-то снаружи. Без этой проверки запрос на удаление файла теоретически мог бы добраться до системных файлов сервера.

## Защита при перезапуске загрузки

Когда загрузку нужно перезапустить, команда yt-dlp читается обратно из сохранённого лога. Перед повторным запуском `Downloader::restart_download()` проверяет, что команда действительно начинается с ожидаемого пути к `yt-dlp` (с учётом возможного префикса `env`). Если кто-то смог бы подменить содержимое лог-файла, эта проверка не даст серверу выполнить произвольную команду из подменённого файла.

### Безопасность авторетрея при гео-блоке

При автоматическом ретрее через прокси (`autoRetryIfNeeded()`) не вводятся новые уязвимости: команда yt-dlp читается из уже сохранённого и проверенного лога, где все параметры защищены через `escapeshellarg()`. Авторетрей просто добавляет флаг `env all_proxy=...` к уже проверенной команде, а сам флаг берётся непосредственно из конфигурации сервера (`config['socks5']`), не от пользователя.

Маркер `[RETRY_ATTEMPTED:timestamp]` в логе файла блокирует повторные попытки, предотвращая потенциальные DoS-атаки через многократное ретрирование одной задачи. Если прокси не настроен в конфиге, авторетрей просто не добавит прокси-флаг (graceful degradation) - задача запустится как была, и пользователь увидит ту же ошибку.

## Санитизация логов

Полные команды запуска yt-dlp, включая параметры прокси с логином и паролем, сохраняются в лог-файлах на сервере. Прежде чем показать содержимое лога пользователю (например, в сообщении об ошибке), `sanitizeLog()` вырезает из текста IP-адреса, доменные адреса с портом, конструкции вида `логин:пароль@хост` и длинные строки, похожие на токены. А при сохранении лога на диск пароль от прокси сразу заменяется плейсхолдером `[SOCKS5_PROXY]` - то есть сам пароль даже не остаётся в открытом виде в файле на сервере.

## Заголовки безопасности

При каждом ответе сервер выставляет набор стандартных защитных заголовков:
- `X-Content-Type-Options: nosniff` - браузер не будет угадывать тип содержимого по содержанию файла.
- `X-Frame-Options: SAMEORIGIN` - страницу нельзя встроить в `<iframe>` на чужом сайте (защита от clickjacking).
- `Content-Security-Policy` - разрешает загружать скрипты и стили только со своего домена.
- `Referrer-Policy` и `Permissions-Policy` - ограничивают утечку данных о посещённой странице и запрещают доступ к камере, микрофону и геолокации, которые сайту вообще не нужны.
- `Strict-Transport-Security` - включается, если соединение защищённое (HTTPS), и заставляет браузер впредь всегда обращаться к сайту только по HTTPS.

## Cookie и сессия

Cookie сессии выставлены с флагами `httponly` (недоступны для чтения через JavaScript - если на сайте всё-таки случится XSS, куки сессии всё равно не украдут) и `samesite=Lax` (браузер не будет автоматически слать эту cookie при запросах с чужих сайтов). Сама PHP-сессия настроена в строгом режиме (`session.use_strict_mode`) и принимает идентификатор сессии только из cookie, никогда из URL.

## Проверка входных данных

URL проверяется через `filter_var(..., FILTER_VALIDATE_URL)` - просто текст без признаков ссылки не пройдёт. Форматы видео и аудио сверяются с жёстким списком разрешённых значений (`sanitizeDlFormat()`, `sanitizeAudioFormat()`) - если пользователь как-то пришлёт значение не из списка, оно тихо заменяется на безопасное значение по умолчанию.

IP-адрес клиента (используется только для записи в лог, кто что скачал) определяется аккуратно: заголовкам `X-Forwarded-For` и `X-Real-IP` доверяют только если сам запрос пришёл с приватного или служебного адреса - то есть точно прошёл через свой nginx или докер-сеть. Если запрос пришёл прямо из интернета (например, порт контейнера случайно открыт напрямую), эти заголовки игнорируются, потому что любой внешний клиент мог бы их подделать.

## Скрытые от веба служебные папки

В конфиге nginx прописан явный запрет (`deny all`) на прямой доступ к папкам `tmp`, `class`, `config` и `.yt_env` - снаружи их вообще не видно, даже если кто-то знает точный путь к файлу.

## Процессы убиваются прицельно

Когда пользователь нажимает "Стоп" на конкретной загрузке, сервер не просит систему "убить процесс с таким PID" вслепую - сначала через `/proc/$pid/cmdline` проверяется, что это действительно процесс `yt-dlp`, а не случайно совпавший по номеру другой процесс на сервере. Это исключает ситуацию, когда из-за переиспользования PID операционной системой команда "стоп" случайно прервёт совершенно другую программу.

## Хранение секретов

Пароль от SOCKS5-прокси никогда не пишется в коде проекта - он живёт в файле `.env`, который не попадает в git, и читается во время выполнения через `getenv()`. В образе Docker это работает за счёт явной настройки `clear_env = no` в конфиге php-fpm - без неё переменные окружения контейнера не дошли бы до PHP-процесса.

## Debug-режим

Параметр `debug` в конфиге по умолчанию выключен. Если его включить, PHP начинает показывать все ошибки и предупреждения прямо на странице - это удобно при разработке, но на сервере с доступом из интернета раскрывает внутренние пути и детали реализации, поэтому в документации к конфигу есть прямое предупреждение не включать его на проде.
