# Починить обход бот-чека YouTube

YouTube иногда требует "доказательство", что запрос идёт от живого человека, а не от скрипта. Без такого доказательства (PO-токена) yt-dlp получает ошибку `HTTP 429` или `Sign in to confirm you're not a bot`, и скачивание не идёт - особенно часто это случается через прокси с уже засвеченным IP.

Качалка обходит это локальным генератором токенов (`bgutil`) - без кук и без аккаунта Google. Обычно это работает незаметно, но иногда ломается, потому что YouTube время от времени меняет правила игры.

## Симптом

В списке загрузок появляется ошибка "YouTube принял нас за бота" - при том, что ещё вчера всё работало.

## Проверить, жив ли сервер токенов

```bash
docker exec yt sh -c 'curl -s 127.0.0.1:4416/ping; echo; tail /var/www/YT/tmp/bgutil.log'
```

Ответ `/ping` должен содержать JSON с версией сервиса. Если ответа нет вообще - сервер токенов внутри контейнера умер (он запускается без авто-рестарта) и его не воскресит ничего, кроме пересборки или перезапуска контейнера.

## Проверить, что плагин загружается

```bash
docker exec yt /.yt_env/bin/yt-dlp --plugin-dirs default -v --simulate 'https://youtu.be/<id>' 2>&1 | grep -i pot
```

Если строк про `pot` нет вообще - значит плагин не подхватился yt-dlp'ом.

## Починить

В большинстве случаев помогает пересборка образа - она подтягивает актуальные версии сразу и плагина, и сервера токенов:

```bash
docker build --no-cache -t yt . && docker-compose up -d
```

## Подключить куки аккаунта (запасной путь)

`bgutil` снимает бот-чек без всякого входа в аккаунт, и в обычной ситуации этого достаточно. Но он не открывает контент, для которого нужен вход: приватные видео, возрастные ограничения, ролики только для подписчиков. А ещё он иногда временно ломается (см. выше). На эти случаи можно подложить настоящие куки залогиненного аккаунта - тогда yt-dlp скачивает так, будто ты открыл ролик в браузере.

Это опция, а не замена `bgutil`. Куки **не** подключаются к каждой загрузке (это только сыпало бы лишние запросы и предупреждения в лог). Первая попытка всегда идёт как обычно, без кук, и только если она падает на бот-чеке, приватности, возрасте или подписке - Качалка сама повторяет ту же загрузку с куками. Пока идёт повтор, во вкладке "Загрузки" видно строку `Обычный способ заблокирован, пробую с куками аккаунта`, а успешный результат помечается в списке значком 🍪.

### Как настроить

1. Заведи **отдельный одноразовый Google-аккаунт**, не личный. Куки на сервере - это, по сути, доступ ко всему аккаунту; если сервер скомпрометируют, потеряется только этот аккаунт.
2. Зайди под ним на youtube.com в браузере и экспортируй куки расширением вида "Get cookies.txt LOCALLY" (формат Netscape, файл `cookies.txt`).
3. Положи файл на сервер вне веб-доступных путей и настрой права (каталог `config` уже закрыт nginx):
   ```bash
   docker cp cookies.txt yt:/var/www/YT/config/youtube_cookies.txt
   docker exec -u root yt chown www-data:www-data /var/www/YT/config/youtube_cookies.txt
   docker exec -u root yt chmod 600 /var/www/YT/config/youtube_cookies.txt
   ```
   Права `600` (или `400`) важны: куки - это доступ ко всему аккаунту, и файл, читаемый группой/остальными, - утечка на мультиюзерном хосте. Руками каждый раз `chmod` делать не нужно: если файл принадлежит `www-data`, код сам закрывает его до `600` при использовании. Ручной `chmod 600` понадобится, только если код не смог этого сделать (файл не принадлежит `www-data` или лежит на ФС без unix-прав, например bind-mount с Windows в dev) - тогда в статусе задачи появится подсказка `Куки настроены, но файл непригоден (права/доступ) - нужен chmod 600`.
4. Пропиши путь в `config.php` (ключ по умолчанию пустой - функция выключена, пока не укажешь путь):
   ```php
   'youtubeCookiesFile' => '/var/www/YT/config/youtube_cookies.txt'
   ```
5. Файл уже в `.gitignore` - в репозиторий не попадёт.

### Когда куки протухнут

Куки живут ограниченное время, особенно если аккаунт заходит только с серверного IP - для Google это подозрительно. Признак: в списке загрузок появляется `Куки YouTube протухли`. Лечится тем же путём - перезайти под тем же аккаунтом, заново экспортировать `cookies.txt`, заменить файл на сервере (права сохранить).

Важно: куки снимают бот-чек и открывают закрытый контент, но **не** лечат рейт-лимит по IP. Если `429` лезет уже на первом запросе страницы - дело в репутации прокси-IP, и никакие куки не помогут (см. ниже).

## Если не помогло

Значит YouTube поменял внутреннюю защиту (BotGuard) сильнее, чем обычно, и нужно подождать, пока разработчики `bgutil` выпустят исправление, а после - пересобрать образ ещё раз.

Отдельно стоит проверить репутацию самого IP-адреса прокси: если ошибка `429` вылезает уже на первом же запросе страницы, это не бот-чек, а обычный рейт-лимит по IP, который никаким обходом не лечится - см. [раздел про бан IP](../explanation/06-youtube-i-perevod-kak-eto-rabotaet.md#что-влияет-на-скорость-бана-ip).

Как всё это устроено внутри - в [Как устроены перевод и обход бот-чека YouTube](../explanation/06-youtube-i-perevod-kak-eto-rabotaet.md).
